3. DATABASE SCHEMA
3.1 Core Tables
Tables Overview:
- users: User accounts and authentication
- trips: Main entity - user's travel trips
- trip_places: Places pinned within trips (user-generated + external)
- trip_routes: GPS tracks or drawn routes between places
- media_files: Photos and videos attached to places
- trip_collaborators: Shared trip editing permissions
- saved_trips: Users saving others' public trips
- user_follows: Social following (for Phase 2)
- subscriptions: Payment and premium tier tracking

3.2 Key Schema Details
users table:
Standard auth fields (email, username, hashed_password)
Profile info (avatar, bio, full_name)
Premium status tracking (is_premium, subscription_ends_at)
Timestamps (created_at, updated_at, last_login)
trips table:
Ownership (user_id foreign key)
Basic info (title, description, start_date, end_date)
Privacy (visibility: private/unlisted/public)
Cover photo (cover_photo_url)
Engagement metrics (views_count, saves_count)
Timestamps
Constraint: end_date >= start_date
trip_places table (Most Important):
Trip association (trip_id foreign key)
Place identification (name, place_type)
Geospatial data (PostGIS Geography type + lat/lng floats)
User content (user_notes, user_rating 1-5, visit_date)
Media arrays (photos JSONB, videos JSONB)
External data cache (external_data JSONB - stores Foursquare/Mapbox data if user saves)
Ordering (order_in_trip integer for itinerary sequence)
Source tracking (to know if from OSM, Foursquare, user-added)
trip_routes table:
Trip association (trip_id foreign key)
Route geometry (PostGIS LineString)
Route metadata (distance_km, duration_minutes, transportation_mode)
Connected places (from_place_id, to_place_id)
Notes about the journey
media_files table:
Ownership (user_id, trip_place_id)
File metadata (file_url, file_type, file_size_bytes, mime_type)
Image specifics (width, height, thumbnail_url)
Content (caption, taken_at timestamp)
Optional location (Geography point if geotagged)
trip_collaborators table:
Trip sharing (trip_id, user_id)
Role (owner/editor/viewer)
Added timestamp
saved_trips table:
User saves (user_id, trip_id)
Saved timestamp
Unique constraint on (user_id, trip_id)
user_follows table (Phase 2 - Social):
Follower relationship (follower_id, following_id)
Created timestamp
Unique constraint, self-follow prevention
subscriptions table:
Payment provider tracking (stripe_customer_id, stripe_subscription_id)
Plan status (plan_type: free/premium, status: active/cancelled/past_due)
Period tracking (current_period_start, current_period_end, cancel_at)
3.3 PostGIS Indexes and Functions
Spatial Indexes (Critical for Performance):
trip_places.location: GIST index for radius searches
trip_routes.path: GIST index for route queries
Full-Text Search Indexes:
trip_places: GIN index on to_tsvector(name + user_notes)
trips: GIN index on to_tsvector(title + description)
Database Functions Required:
update_updated_at_column(): Auto-update timestamp trigger
calculate_trip_bounds(trip_id): Get bounding box for map viewport
nearby_places(lat, lng, radius_km): Find places within radius
calculate_route_distance(linestring): Get route length in km

5. SERVICE LAYER ARCHITECTURE
5.1 Multi-Source Search Service (Critical Component)
Purpose: Search for places using multiple APIs while respecting ToS
Search Strategy:
1. Query local database (user-contributed places from trip_places table)
   - Use PostGIS ST_DWithin for radius search
   - Use PostgreSQL full-text search for name/notes
   - Prioritize places with user_rating > 0
   
2. If < 5 results, query Foursquare Places API
   - Use /v3/places/search endpoint
   - Pass query, lat/lng, radius
   - Map response to common PlaceResult format
   
3. If still < 5 results, query Mapbox Search API
   - Use geocoding/v5/mapbox.places with type=poi
   - Map response to common PlaceResult format
   
4. Optionally query Nominatim (OSM)
   - Use /search endpoint with format=json
   - Good fallback for small towns/villages
   
5. Deduplicate results
   - Fuzzy match on (name.lower(), round(lat, 3), round(lng, 3))
   - Keep highest quality source (user > foursquare > mapbox > osm)
   
6. Rank results
   - Priority: source type (user=0, foursquare=1, mapbox=2)
   - Secondary: user_rating DESC
   - Tertiary: distance ASC
   
7. Return top 10 results with source attribution

PlaceSearchResult Schema:
{
  "id": "UUID or external_id",
  "name": "string",
  "place_type": "string",
  "lat": "float",
  "lng": "float",
  "address": "string?",
  "source": "user_contributed | foursquare | mapbox | osm",
  "distance_km": "float",
  "user_rating": "int? (1-5, only for user_contributed)",
  "has_user_content": "bool (photos/notes from users)",
  "can_claim": "bool (external sources can be added to user's DB)",
  "external_id": "string? (foursquare fsq_id, mapbox id)"
}

5.5 Route Service
Responsibilities:
Create routes from GPS tracks or manual drawing
Calculate route via OSRM or Mapbox Directions
Store route geometry in PostGIS
Calculate distance and duration
Routing Options:
Option A: Self-hosted OSRM
  - Download OSM data for India + Nepal
  - Process with osrm-extract and osrm-contract
  - Run osrm-routed on VPS
  - Cost: $10/month VPS
  - Latency: Fast (self-hosted)
  
Option B: Mapbox Directions API
  - Use /directions/v5 endpoint
  - Cost: Free tier 100k requests/month
  - Easier setup, no maintenance
  
Recommendation for MVP: Option B (Mapbox)
Switch to Option A (OSRM) only if you exceed free tier

Route Calculation Flow:
1. Receive from_lat/lng and to_lat/lng
2. Call routing API (OSRM or Mapbox)
3. Get back route geometry (array of [lng, lat] coordinates)
4. Calculate distance and duration
5. Return to frontend for display
6. If user saves, store in trip_routes table as PostGIS LineString

## 9. TECHNICAL CONSTRAINTS & REQUIREMENTS

### 9.1 Performance Requirements

**API Response Times:**
- Simple queries (GET user, trip): < 100ms
- Place search (multi-source): < 500ms
- Map tile requests: < 200ms (handled by Mapbox CDN)
- Photo upload: < 3s for 5MB file

**Database Query Optimization:**
- All geospatial queries must use PostGIS spatial indexes
- Full-text search must use GIN indexes
- Pagination required for all list endpoints (max 100 items per page)
- Use SELECT only needed columns (no SELECT *)

**Frontend Performance:**
- Initial page load: < 2s
- Map render: < 1s
- Route to route transition: < 300ms
- Lazy load images (intersection observer)
- Code splitting for routes

### 9.2 Security Requirements

**Authentication:**
- Passwords hashed with bcrypt (cost factor 12)
- JWT tokens with 24h expiration
- Refresh tokens with 30-day expiration
- HttpOnly cookies for refresh tokens (CSRF protection)
- Rate limiting on auth endpoints (5 attempts per 15 min)

**Authorization:**
- Row-level security checks on all database queries
- Never trust client-side data
- Validate all inputs with Pydantic
- SQL injection prevention (use parameterized queries)

**File Upload Security:**
- Validate file types (magic number check, not just extension)
- Virus scanning (ClamAV integration for production)
- Size limits enforced server-side
- Unique filenames (no user-controlled names)
- Separate storage bucket for user uploads (not code)

**API Security:**
- CORS configured for specific origins only
- Rate limiting (100 requests/min per user)
- API keys for external services stored in environment variables
- No sensitive data in logs
- HTTPS only in production

### 9.3 Data Privacy & Compliance

**User Data:**
- Users can export all their data (GDPR compliance)
- Users can delete their account and all data
- Clear privacy policy explaining data usage
- No selling of user data
- Anonymous usage analytics (opt-out available)

**Geolocation:**
- GPS data only with explicit user permission
- Can disable location services in settings
- Location data never shared publicly without consent

**Photos/Videos:**
- EXIF data stripped on upload (remove GPS, camera model)
- User owns all their content
- Private trips are truly private (not indexed, not discoverable)

### 9.4 Scalability Considerations

**Database:**
- PostgreSQL with connection pooling (max 20 connections for MVP)
- Read replicas for scaling reads (future)
- Partitioning trip_places table by created_at (after 1M rows)

**Caching:**
- Redis for session storage
- API response caching (public trips, user profiles)
- Cloudflare CDN for static assets and images
- Map tiles cached by Mapbox

**File Storage:**
- R2 auto-scales (no limits)
- CDN in front of R2 for faster delivery
- Lazy loading images on frontend

**API:**
- Horizontal scaling with load balancer (future)
- Stateless API (no server-side sessions)
- Background jobs in Celery for heavy tasks
